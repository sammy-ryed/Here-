--------------------------------------------------------------------------------
  WHAT IS THIS?
--------------------------------------------------------------------------------

An automated classroom attendance system that uses face recognition to mark
students present directly from a photograph of the classroom. Faculty uploads
one (or more) photos of the class; the system detects every face in the image,
matches each face against registered students, and records attendance — all
within seconds. No manual roll-call, no proxy attendance.

There is also a live webcam mode for 1-by-1 recognition and a full self-
registration portal so faculty never has to handle student photos.

--------------------------------------------------------------------------------
  FULL FEATURE LIST
--------------------------------------------------------------------------------

1. STUDENT REGISTRATION (Manual)
   - Faculty enters Name + Roll No, uploads 5-8 photos of the student
   - Backend processes images in parallel (ThreadPoolExecutor)
   - RetinaFace detects the face in each photo; Facenet512 generates a 512-dim
     embedding per image
   - All embeddings are averaged into a centroid and stored in SQLite
   - Individual embeddings are also stored for centroid rebuilding later

2. BULK IMPORT (Excel upload)
   - Faculty uploads a .xlsx file
   - Auto-detects one of two modes based on column headers:

   MODE A — SELF-REGISTRATION (no Drive Link column)
     * Creates a pending student record (no photos yet)
     * Generates a secure random token per student (expires in 7 days)
     * Faculty distributes the links (copy/CSV download)
     * Students open their link, submit their own photos via webcam or upload
     * Backend processes photos, stores embeddings, marks token used
     * Student is now fully registered without faculty touching photos

   MODE B — DRIVE LINKS (Drive Link column present)
     * Downloads each student's Google Drive photo folder via gdown (up to 8 imgs)
     * Runs full face detection + embedding on each image
     * Registers the student immediately, no tokens needed
     * If a student's Drive Link cell is blank / "NA" / "N/A" / "none" / "nil" /
       "-" etc., the system automatically falls back to Mode A for that student

   - Mixed results: registered-via-drive students + token students are both
     shown in separate tables after import

3. ATTENDANCE — PHOTO UPLOAD (main mode)
   - Faculty uploads 1-N classroom photos
   - Single photo -> /process_attendance
   - Multiple photos -> /process_attendance_batch (merges results, deduplicates)
   - For each photo:
     * RetinaFace detects all faces
     * Each face is processed in parallel (2 threads)
     * Facenet512 generates embedding
     * Matrix cosine similarity search against all registered students
     * Threshold 0.769 — match is accepted only above this value
     * Matched students -> marked PRESENT with confidence score
     * Unmatched faces -> saved to /unrecognized/ folder for manual review
   - Result table: Present / Absent / Unrecognized counts + per-student detail
   - Absent students shown with last-seen date

4. ATTENDANCE — LIVE WEBCAM
   - Real-time face recognition from browser webcam
   - Sends a JPEG frame to /recognize/face every 1 second
   - Recognised students accumulate in a "seen today" list
   - Shows FPS counter + current detection overlay
   - Good for entrance / corridor scanning

5. ADD EMBEDDINGS (Embeddings page — 2-step flow)
   - Step 1: Upload any group/classroom photo -> /extract_faces
     Backend crops every detected face, saves temp files, returns base64 thumbnails
   - Step 2: Face thumbnail grid shown on screen
     Faculty assigns each face to a student via dropdown, clicks Assign
     -> /assign_face/<student_id> with face_filename (JSON)
     Backend computes embedding from the crop, stores it, rebuilds the index
   - Skip button to ignore unknown faces
   - Each assigned face immediately improves recognition for that student
   - Support for direct photo upload per student also exists via /add_embedding

6. ATTENDANCE RECORDS & DASHBOARD
   - Dashboard: total students, present today, absent today, attendance rate %
   - Attendance page: date-filtered view, manual mark present/absent per student
   - Confidence score stored per attendance record

7. STUDENT MANAGEMENT
   - View all students with has_embedding flag
   - Delete student (cascades to embeddings + attendance)
   - Profile fields: Name, Roll No, Section, Course, Department, Room No


--------------------------------------------------------------------------------
  HOW FACE RECOGNITION WORKS (DEEP DIVE)
--------------------------------------------------------------------------------

DETECTION (RetinaFace)
  - RetinaFace is a joint face detection + landmark localisation model
  - It returns bounding boxes, confidence scores, and 5-point facial landmarks
  - Smart quality check: if image is blurry (Laplacian variance < 100) or
    poorly lit (brightness outside 100-180), it applies CLAHE enhancement
    before detection, otherwise runs raw for speed

EMBEDDING (Facenet512)
  - Each detected face is cropped with a 30px margin, saved as temp JPEG
  - DeepFace.represent() is called with detector_backend='skip' (detection
    already done by RetinaFace), align=False for speed
  - Returns a 512-dimensional float32 vector
  - Vector is L2-normalised immediately (so dot product == cosine similarity)
  - Cached by (image_path + bbox hash) — repeated requests are instant

MATCHING (Matrix search)
  - At startup (and after any registration/embedding add), build_student_index()
    runs: for each student, all their stored embeddings are fetched, averaged
    into a centroid, and re-normalised
  - Result: a single (N_students x 512) float32 matrix in memory
  - At recognition time: query embedding dot-multiplied against full matrix
    in one BLAS call -> O(N x 512) but vectorised, effectively O(1) for
    typical class sizes
  - Highest cosine similarity student wins if score >= 0.769
  - Thread-safe: a lock prevents two threads from marking the same student
    twice from the same image

CONFIDENCE THRESHOLD: 0.769
  - Cosine similarity ranges from -1 to 1; 1 = identical
  - 0.769 was tuned to balance false-rejects vs false-accepts in classroom
    conditions (variable lighting, angles, occlusion)
  - Raising it reduces false positives but may miss valid matches
  - Lowering it catches more students but risks proxy attendance

CENTROID AVERAGING
  - Multiple photos per student -> multiple embeddings
  - The centroid (average + re-normalise) represents the student's "average"
    appearance across lighting / pose variations
  - More photos = more robust centroid = higher accuracy


--------------------------------------------------------------------------------
  ACCURACY
--------------------------------------------------------------------------------

Facenet512 achieves ~99.65% accuracy on the LFW (Labelled Faces in the Wild)
benchmark under controlled conditions.

In real classroom conditions accuracy depends on:
  - Number of registration photos per student (5-8 recommended)
    * 3 photos:   ~85-90% recognition rate
    * 5-8 photos: ~93-97% recognition rate
  - Image quality (resolution, blur, lighting)
  - Face visibility (angle, occlusion by hair/glasses)
  - Class size: larger classes slightly slow down processing but do not
    affect accuracy (matrix search is vectorised)

Threshold 0.769 was chosen to minimise proxy attendance fraud while keeping
the false-reject rate acceptable under normal classroom lighting.

Unrecognised faces are saved to /unrecognized/ for manual review, so the
faculty can catch any missed students and manually mark them via the dashboard.

--------------------------------------------------------------------------------
  ADVANTAGES
--------------------------------------------------------------------------------

1. FAST — a 30-student classroom photo processes in ~5-15 seconds end-to-end
2. NON-INVASIVE — no fingerprint scanner, no RFID card, works from a distance
3. SCALABLE — adding more students only grows the matrix; matching is O(1) per query
4. PRIVACY-FRIENDLY — students upload their own photos via token links;
   faculty never handles personal photos
5. MULTI-PHOTO ROBUSTNESS — centroid averaging across 5-8 photos handles
   lighting/pose variation much better than a single photo
6. FORGERY-RESISTANT — proxy attendance is essentially impossible;
   the system checks actual facial geometry, not a card or name
7. OFFLINE-CAPABLE — runs entirely on local hardware; no cloud API keys needed
8. MANUAL OVERRIDE — faculty can manually mark any student on the dashboard
9. UNRECOGNISED FACE LOGGING — missed students are saved as crops for review
10. INCREMENTAL IMPROVEMENT — add more face photos anytime via the embeddings
    page; the centroid immediately updates for the next attendance session
11. ZERO-DATA-LOSS MIGRATION — schema evolves with ALTER TABLE IF NOT EXISTS,
    safe to update without wiping existing student/attendance data


--------------------------------------------------------------------------------
  DRAWBACKS / LIMITATIONS
--------------------------------------------------------------------------------

1. FIRST-RUN LATENCY — DeepFace downloads the Facenet512 model weights
   (~90 MB) on first run; subsequent starts are instant

2. CPU-BOUND — without a GPU, processing 30+ faces takes 10-30 seconds;
   with a CUDA GPU it drops to 2-5 seconds. TensorFlow will use GPU if available.

3. POOR LIGHTING — very dark, backlit, or blown-out images can drop accuracy
   significantly. Smart CLAHE enhancement helps but is not perfect.

4. OCCLUSION — heavy masks, sunglasses, or extreme head angles (>45 degrees)
   can cause missed detections

5. IDENTICAL TWINS — cosine similarity cannot distinguish identical twins
   reliably (fundamental biometric limitation, not fixable in software)

6. SINGLE THRESHOLD — the 0.769 threshold is global; students who always
   photograph poorly may need more registration photos, and the threshold
   cannot be tuned per-student without code changes

7. SQLite CONCURRENCY — WAL mode handles light concurrent load well, but
   under heavy simultaneous access (10+ threads writing) it can bottleneck;
   a production deployment should use PostgreSQL

8. NO AUTHENTICATION — the admin frontend has no login/auth; it assumes
   trusted local or intranet access only. Exposing to the public internet
   as-is is a security risk.

9. TOKEN CLEANUP — expired tokens are checked at use time but are never
   automatically purged from the DB (no background cleanup job)

10. NO ATTENDANCE HISTORY EXPORT — the frontend shows today's attendance and
    a records query, but there is no calendar heatmap or export-to-Excel of
    historical attendance yet

11. SELF-REGISTRATION LINK DELIVERY — token links are generated but the system
    has NO email/SMS integration; faculty must distribute them manually (copy/CSV)

12. DRIVE MODE RATE LIMITS — gdown can hit Google Drive download quotas for
    large batches; Drive folders must be set to "Anyone with the link can view"


--------------------------------------------------------------------------------
  SECURITY NOTES
--------------------------------------------------------------------------------

- Self-registration tokens: 32-byte cryptographically random (secrets.token_urlsafe)
- Tokens are single-use; once a student registers, the token is marked used
- Token expiry is enforced server-side (7 days, checked on every call)
- File uploads validated by extension (png/jpg/jpeg only) before any processing
- No authentication on admin frontend — intended for intranet / localhost use only
- CORS is fully open (Flask-CORS *) — restrict origins in production

Project Overview:
-----------------
This system automates attendance using face recognition. The JavaFX frontend captures photos or live webcam frames, detects faces, and sends them to the Python Flask backend. The backend uses DeepFace (Facenet512 for embeddings, RetinaFace for detection) to recognize faces and mark attendance in the SQLite database. All embeddings are preloaded for fast comparison, and the system is optimized for speed with batch processing, threading, and direct numpy operations. Attendance is marked automatically when a face is recognized, and results are shown in real time.

Possible Critical Questions & Answers:
--------------------------------------
Q1: How do you ensure fast and accurate face recognition?
A1: We use DeepFace with Facenet512 for high-accuracy embeddings, batch process faces, preload all embeddings in RAM, and use direct numpy for fast cosine similarity. Threading and WAL-mode SQLite further speed up processing.

Q2: How do you handle multiple faces in one frame or photo?
A2: The backend detects all faces in a photo/frame, extracts embeddings for each, and matches them in parallel. Attendance is marked for every recognized face.

Q3: What if two students look very similar?
A3: We use a high-dimensional (512D) embedding and a strict similarity threshold to minimize false positives. Only high-confidence matches are accepted.

Q4: How do you prevent duplicate attendance?
A4: The backend checks if a student is already marked present for the day before updating the database, ensuring no duplicate entries.

Q5: What happens if the database is locked or busy?
A5: The system uses SQLite WAL mode and retry logic to handle concurrent access and avoid lock errors, ensuring reliability even under heavy load.

Q6: Can this system work offline?
A6: Yes, all processing is local. No internet is required after setup.

Q7: How secure is the data?
A7: All face data and attendance records are stored locally in SQLite. No data is sent to third parties. Only embeddings (not raw images) are stored for privacy.
